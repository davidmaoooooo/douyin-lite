# 给逆向 agent 的补充素材（主 Claude 提供，基于浏览器实验）

## 1. hook 明文捷径失败
- hook `String.fromCharCode` → count=0，JSVMP 不走原生 fromCharCode。
- 明文捷径需 hook JSVMP 解释器 `_$webrt_1668687510`，等同读字节码，暂不可行。
- 结论：只能靠密文差分恢复，你的密码学攻击路线是对的。

## 2. 旧版 generate_random_str 在固定 Math.random 下的确定字节
旧版逻辑：`gener_random(Math.random()*10000,[3,45]) ++ gener_random(...,[1,0]) ++ gener_random(...,[1,5])` = 12字节
gener_random(r,opt) = [
  (r&255&170)|(opt0&85), (r&255&85)|(opt0&170),
  ((r>>8)&255&170)|(opt1&85), ((r>>8)&255&85)|(opt1&170)
]

| Math.random | *10000取整 | 12字节随机串 |
|---|---|---|
| 0.0  | 0    | [1, 2, 5, 40, 1, 0, 0, 0, 1, 0, 5, 0] |
| 0.25 | 2500 | [129, 70, 13, 41, 129, 68, 8, 1, 129, 68, 13, 1] |
| 0.5  | 5000 | [137, 2, 7, 57, 137, 0, 2, 17, 137, 0, 7, 17] |
| 0.75 | 7500 | [9, 70, 13, 61, 9, 68, 8, 21, 9, 68, 13, 21] |

**注意**：新版改 random 导致整段雪崩，说明随机串可能进了 RC4 输入（与旧版"明文拼接"不同）。
若新版随机串 = 上表字节，且它是 RC4 明文前12字节，则：
  对同一密钥, cipher[i] = plain[i] XOR keystream[i]
  用 random_dim 两个样本(如0.0 vs 0.5): cipher_A[i]^cipher_B[i] = plain_A[i]^plain_B[i]（keystream抵消）
  → 可验证随机串字节是否吻合（若解码字节差 == 上表字节差，则假设成立，且能恢复keystream）

**但前提存疑**：新版随机串逻辑/长度可能已变（旧版12字节，新版前缀更长）。需先验证。
