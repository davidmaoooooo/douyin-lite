# a_bogus 硬逆向 - 中场结论（务实评估）

## 已**确定**的成果（实测+脚本验证）
1. 整体结构：`a_bogus = s4_encode(明文137字节) + "="`，s4 表与旧版一致。
2. 明文 137 字节 = random_head + 主体(经SM3/RC4) + 尾部校验。
3. **确定性可控**：冻结 Date.now/Math.random/performance.now 三源后，同输入产出完全一致（连续5次/重复验证）。
4. **可靠采集器**：浏览器内用 SDK 重写的 XHR 发请求取 responseURL 的 a_bogus。
5. query 注入位置：明文字节 24,27,28,31,32,35（单字符query只动24,28,32,35）+ 尾136校验。
6. time 注入位置：低位 idx16,18,19；另一段 idx39,41,43,46；idx70,71；校验133,135,136。
7. random_head 前段是 gener_random 结构，option 常量已现形 [3,82],[117,34],[184,169]...（旧版是[3,45],[1,0],[1,5]）。
8. method(GET=POST)/body/performance.now 不影响结果。
9. 固定模板常量字节：idx 15=221, 19=124, 23=52, 39=178（与所有可变输入无关）。

## 关键障碍（决定可行性）
**random 影响穿透整个明文**（改 random，idx 0~38 绝大多数字节都变，只有极少数固定）。
→ random 字节既是 random_head 明文，又**进入了后续 SM3/RC4 的输入**，造成全局雪崩。
→ 变量高度耦合：无法像旧版那样把 query/time/ua/env 各字段独立隔离。
→ RC4 密钥很可能是**动态派生**（依赖 random/time），不是固定常量，所以已知明文攻击难以恢复稳定 keystream。

## 可行性判断
- 黑盒差分能定位"哪些字节受哪些输入影响"（已做到），但**无法在变量耦合下反推出加密前的字节构造公式**——因为 SM3 是单向哈希，RC4 密钥动态，明文模板未知，三者叠加。
- 要真正还原，唯一可靠路径是**读 JSVMP 字节码**（反编译 `_$webrt_1668687510` 解释器 + dump 字节码指令流），这是数周级专业逆向工作，且抖音更新即失效。
- 纯黑盒继续投入：成功概率低，性价比差。

## 已排除的捷径
- hook String.fromCharCode 抓明文：JSVMP 不走原生函数，失败。
- RC4 key=121 解密：密钥已变，失败。
